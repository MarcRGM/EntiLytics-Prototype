# System Architecture

EntiLytics is a single-process Python web application built with the Solara framework. There is no REST API layer. All components, the UI, the NLP pipeline, and the database, run within the same Python process. Communication between them happens through direct function calls and Solara reactive variables.

## Application Layer (app.py)

**Framework:** Solara (Python reactive UI)

The entire application lives in `app.py`. Solara renders the UI by re-running Python component functions whenever a reactive variable changes. The browser communicates with the Solara server over a persistent WebSocket connection managed by Solara internally.

### Reactive State

All application state is held in `solara.reactive` variables at module level. When a variable's value changes, Solara re-renders only the components that depend on it.

| Variable                | Type           | Purpose                                               |
| ----------------------- | -------------- | ----------------------------------------------------- |
| `current_user`          | `dict or None` | Holds logged-in user profile from Google OAuth        |
| `current_session_id`    | `str or None`  | UUID of the active session stored in the database     |
| `is_checking_session`   | `bool`         | Loading gate while session restoration is in progress |
| `selected_article_data` | `dict or None` | Holds the full NLP output for the result view         |
| `rss_feed_results`      | `list`         | List of article metadata fetched from an RSS feed     |
| `input_mode`            | `str`          | Switches the input form between `manual` and `rss`    |
| `is_loading`            | `bool`         | Controls the progress bar during NLP processing       |
| `save_status`           | `str`          | Triggers sidebar refresh and success/error messages   |

### Page Routing

The `Page` component acts as a traffic controller. It renders `LoginScreen` when `current_user` is `None` and `DashboardScreen` when the user is logged in. There are no URL-based routes for different views. All navigation is driven by reactive variable changes.

```python
if current_user.value is None:
    LoginScreen()
else:
    DashboardScreen()
```

## Authentication (auth_handler.py)

**Protocol:** Google OAuth 2.0 (Authorization Code Flow)

Login uses the standard three-step OAuth 2.0 flow:

1. The user clicks the login button, which redirects the browser to Google's authorization URL generated by `get_google_login_url()`
2. Google redirects back to the application with a `?code=` query parameter
3. The `LoginScreen` component detects the code in the URL, calls `exchange_code_for_user_info()` to trade the code for an access token, then uses that token to fetch the user's email, name, and profile picture from Google's user info endpoint

After a successful login, a UUID session ID is created and stored in both the `user_sessions` database table and the browser as a cookie named `entil_sid` with a 7-day expiry.

### Session Problem on Page Refresh

When the page loads, the `SessionRestorer` component injects a JavaScript snippet that reads the `entil_sid` cookie and passes it back to Python via a hidden Solara input field (the "bridge"). Python then queries the `user_sessions` table to verify the session and restore `current_user` if valid.

There is a known timing issue: Solara renders the UI before the JavaScript bridge completes, which can cause the login screen to appear on refresh even for authenticated users. The `is_checking_session` reactive variable is used as a loading gate to address this, but the behavior is under active development.

## NLP Pipeline

All NLP processing is triggered by direct Python function calls within `app.py`. The pipeline runs sequentially when a user submits an article for analysis:

```
analyze_article()
     │
     ├── identify_entities()       (flair_ner.py)
     ├── entity_ranking()          (entity_ranking_and_summarization.py)
     ├── generate_summary()        (entity_ranking_and_summarization.py)
     └── mapping()                 (relationship_mapping.py)
```

### Step 1: Entity Extraction (flair_ner.py)

**Model:** Flair `ner` (BiLSTM pretrained on CoNLL-2003)

The article text is passed to Flair's `Classifier.load('ner')` model. Flair tokenizes the text internally, runs it through the BiLSTM architecture in both the forward and backward directions, concatenates the outputs, and applies a softmax layer to classify each token with a BIO entity label. Consecutive B- and I- tags are grouped into complete named entity spans.

Each entity is returned as a dictionary containing the entity text, its label (`PER`, `ORG`, `LOC`, or `MISC`), and a confidence score.

### Step 2: Entity Importance Ranking (entity_ranking_and_summarization.py)

**Model:** `all-MiniLM-L6-v2` (SentenceTransformers, BERT-based)

The extracted entities and the full article text are each encoded into 384-dimensional BERT vectors. Vectors are L2-normalized, then **Manhattan distance** is computed between each entity vector and the article vector.

A lower distance means the entity is semantically closer to the article as a whole, indicating higher importance. Entities are filtered by a threshold of `0.45` (based on findings from research on BERT-based paraphrase detection using the MRPC dataset) and sorted in ascending order of distance. The result is a ranked list of entity name and distance score pairs.

The distance is normalized by dividing by `2 * sqrt(384)` (approximately 39.2) to produce a consistent 0.0 to 1.0 scale across articles of different lengths.

### Step 3: Extractive Summarization (entity_ranking_and_summarization.py)

**Model:** Same `all-MiniLM-L6-v2` model

The article is split into sentences using NLTK's `sent_tokenize`. The top-ranked entity names are joined into a single string and encoded into a BERT vector. Each sentence is also encoded, and Manhattan distance is computed between each sentence vector and the entity string vector.

Sentences within the `0.45` threshold are selected, re-ordered by their original position in the article, and joined to form the final summary. This extractive approach avoids entity hallucination, a known failure mode where abstractive models generate entity names or facts not present in the source text.

### Step 4: Relationship Mapping (relationship_mapping.py)

**Libraries:** NetworkX, Pyvis, BeautifulSoup, NLTK

The article is split into sentences using NLTK. For each sentence, the system checks which of the top-ranked entity names appear in the sentence text. When two or more entities co-occur in the same sentence, an edge is created between them in a NetworkX graph. Repeated co-occurrences accumulate as evidence sentences on the same edge.

Pyvis renders the NetworkX graph as an interactive HTML string. Each entity is a yellow box node. Edge thickness reflects co-occurrence frequency, and hovering over an edge shows the sentences where the pair appeared together. The HTML is stored in the database and embedded in the results view inside a Solara `iframe`.

## RSS Feed Handling (rss_handler.py)

**Library:** feedparser

When the user provides an RSS feed URL, `fetch_rss_articles()` parses the feed and returns a list of article metadata dictionaries. Each entry contains the title, description, link, publication date, and source URL. The description field serves as the article text for NLP analysis after HTML tags are stripped by BeautifulSoup in `analyze_article()`.

## Database Layer (database.py)

**Technology:** SQLAlchemy ORM, PostgreSQL (Azure Flexible Server)

All database access goes through SQLAlchemy sessions, opened and closed per operation. The connection string requires SSL (`sslmode=require`) and uses a 10-second connection timeout.

### ORM Models

| Model            | Table             | Description                                               |
| ---------------- | ----------------- | --------------------------------------------------------- |
| `Account`        | `account`         | Registered users identified by Google email address       |
| `UserSession`    | `user_sessions`   | Active sessions with UUID session ID and 7-day expiry     |
| `Article`        | `article`         | Article title and full cleaned text content               |
| `Summary`        | `summary`         | Extractive summary text linked to an article and user     |
| `AnalysisResult` | `analysis_result` | Rankings JSON, entities JSON, and relationship graph HTML |
| `Annotation`     | `annotation`      | User-written notes linked to an article and user account  |

### Key Database Functions in app.py

| Function                        | Description                                                              |
| ------------------------------- | ------------------------------------------------------------------------ |
| `sync_user_to_db()`             | Creates an `Account` record on first login if one does not exist         |
| `create_session()`              | Inserts a `UserSession` row and returns the new UUID                     |
| `resolve_session()`             | Looks up a session by UUID, checks expiry, returns user info             |
| `save_to_azure()`               | Upserts `Article`, `Summary`, `AnalysisResult`, and `Annotation` records |
| `get_saved_titles()`            | Returns all articles linked to the current user via their summaries      |
| `display_historical_analysis()` | Fetches saved results and restores them into `selected_article_data`     |
| `delete_current_article()`      | Cascade-deletes all records linked to a given article ID                 |

## Deployment

The application is containerized using Docker and deployed to Azure App Service for Containers. The container runs the Solara server bound to `0.0.0.0:8080`. Azure App Service maps external ports 80 and 443 to the container's internal port 8080 via the `WEBSITES_PORT=8080` application setting.

Every push to the `main` branch triggers a GitHub Actions workflow that builds a new Docker image, pushes it to Azure Container Registry, and instructs Azure App Service to pull and restart with the new image.

For full deployment documentation, see [`deployment_guide.ipynb`](deployment_guide.ipynb).
